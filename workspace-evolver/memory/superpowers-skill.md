# Superpowers Skill 使用指南

> Evolver 整理的 Superpowers skill 知识

---

## 1. Superpowers 概述

**Superpowers** 是一个规范优先、TDD、子智能体驱动的软件开发工作流 skill。

### 1.1 核心流程

```
Idea → Brainstorm → Plan → Subagent-Driven Build (TDD) → Code Review → Finish Branch
```

**关键原则**：
- "太简单不需要设计" 永远是错的
- 在用户批准设计之前不得编写任何代码
- 每个编码任务都遵循此流程

### 1.2 适用场景

| 场景 | 触发 |
|-----|------|
| 构建新功能/应用 | → Brainstorm → Plan → Subagent 执行 |
| 调试 Bug 或测试失败 | → 系统性根因分析流程 |
| 用户说 "let's build", "help me plan", "I want to add X" | → Brainstorm |
| 完成功能分支 | → 测试验证 + 合并/PR 选项 |

**不适用于**：
- 单行修复（直接编辑）
- 阅读代码
- 非编码任务

## 2. Phase 1: Brainstorming（头脑风暴）

### 2.1 触发条件

用户想要构建某些东西。在接触任何代码之前激活。

### 2.2 检查清单（按顺序）

1. **探索项目上下文** — 检查文件、文档、最近提交
2. **提出澄清问题** — 一次一个，偏好多选题
3. **提出 2-3 种方法** — 带权衡和推荐
4. **展示设计** — 按复杂度分节，每节后获得批准
5. **编写设计文档** — `docs/plans/YYYY-MM-DD-<topic>-design.md` → 提交
6. **过渡** — 调用 writing-plans 阶段

### 2.3 规则

- 每条消息只问一个问题
- 多选题优于开放式问题
- 每个项目都经过此流程 — "简单"项目也不例外
- **硬性门槛**：在设计批准之前不得编写代码、脚手架或实现任何东西
- 在确定方案前提出 2-3 种方法，首先给出推荐

### 2.4 要问的问题

| 问题 | 目的 |
|-----|------|
| 你真正想做什么？ | 目的 |
| 存在哪些约束？ | 时间、技术栈、依赖 |
| 成功是什么样的？ | 成功标准 |
| 这不应该做什么？ | 范围边界 |

### 2.5 设计要覆盖的部分

- 架构概述
- 组件及其职责
- 数据流
- 错误处理方法
- 测试策略

### 2.6 设计批准后

1. 将设计写入 `docs/plans/YYYY-MM-DD-<topic>-design.md`
2. 提交设计文档
3. 移交给 writing-plans — 不调用其他 skill，不实现

---

## 3. Phase 2: Writing Plans（编写计划）

### 3.1 触发条件

设计已批准。由 brainstorming 阶段激活。

### 3.2 任务要点

- 编写详细的逐任务实现计划
- 每个任务 = 2-5 分钟：写测试 → 看失败 → 实现 → 看通过 → 提交
- 保存到 `docs/plans/YYYY-MM-DD-<feature>.md`
- 宣布：`"I'm using the writing-plans skill to create the implementation plan."`

### 3.3 执行模式选择

保存计划后，提供两种执行模式：

| 模式 | 说明 |
|-----|------|
| **子智能体驱动（当前会话）** | 每个任务 `sessions_spawn` + 两阶段审查 |
| **手动执行** | 用户自己运行任务 |

---

## 4. Phase 3: Subagent-Driven Development（子智能体驱动开发）

### 4.1 触发条件

计划存在，用户选择子智能体驱动执行。

### 4.2 每任务循环

```
1. sessions_spawn 实现子智能体（任务 + 完整计划上下文）
2. 等待完成公告
3. sessions_spawn 规格审查子智能体 → 必须确认代码符合规格
4. sessions_spawn 代码质量审查子智能体 → 必须批准质量
5. 修复任何问题，必要时重新审查
6. 标记任务完成，移动到下一个
7. 最终：派发整体代码审查员 → 移交 Phase 5
```

### 4.3 子智能体派发模式

```
Goal: [一句话]
Context: [为什么重要，哪个计划文件]
Files: [确切路径]
Constraints: [不要做什么 — 不蔓延范围，仅 TDD]
Verify: [如何确认成功 — 测试通过，具体命令]
Task text: [粘贴计划中的完整任务]
```

### 4.4 TDD 是强制性的

每个任务都必须遵循 TDD：
1. 写失败的测试
2. 运行测试，确认失败
3. 实现最小代码使测试通过
4. 运行测试，确认通过
5. 提交

---

## 5. Phase 4: Systematic Debugging（系统性调试）

### 5.1 触发条件

Bug、测试失败、意外行为 — 任何技术问题。

### 5.2 硬性门槛

**没有根因调查就不能修复。**

### 5.3 四个阶段

| 阶段 | 活动 |
|-----|------|
| **1. 根因调查** | 阅读错误、复现、检查最近更改、追踪数据流 |
| **2. 模式分析** | 找到工作示例、比较、识别差异 |
| **3. 假设 + 测试** | 一次一个假设，测试证明/反驳 |
| **4. 修复 + 验证** | 在根因处修复，不是症状；验证修复不破坏任何东西 |

---

## 6. Phase 5: Finishing a Branch（完成分支）

### 6.1 触发条件

所有任务完成，所有测试通过。

### 6.2 步骤

1. 验证所有测试通过
2. 确定基础分支
3. 提供 4 个选项：
   - 本地合并
   - 推送 + PR
   - 保留
   - 丢弃
4. 执行选择
5. 清理

---

## 7. 关键原则

| 原则 | 说明 |
|-----|------|
| **一次一个问题** | 头脑风暴期间 |
| **TDD 永远** | 先写失败的测试，删除测试前写的代码 |
| **YAGNI** | 从所有设计中删除不必要的功能 |
| **DRY** | 不重复 |
| **系统化优于临时** | 特别是在时间压力下遵循流程 |
| **证据优于声明** | 声明成功前验证 |
| **频繁提交** | 每个绿色测试后 |

## 8. 如何应用于 AI 学习和 OpenClaw

### 8.1 Brainstorming 扩展思路

使用 Superpowers 的 brainstorming 方法来：

1. **探索上下文**
   - 当前知识库有什么？
   - 还有哪些文档需要学习？
   - 最近的学习重点是什么？

2. **提出澄清问题**（一次一个）
   - 你想深入哪个具体领域？
   - 你的学习目标是什么？（理论 vs 实践）
   - 你有多少时间投入？

3. **提出 2-3 种方法**
   - 方法 A：深度学习某框架
   - 方法 B：广泛学习多个框架
   - 方法 C：实践项目驱动学习

4. **展示学习计划**
   - 分节展示，每节后确认
   - 获得批准后执行

### 8.2 应用于学习任务

```
学习目标 → Brainstorm（探索领域、提问、提方案）
         → Plan（制定详细学习计划）
         → Execute（子智能体执行研究任务）
         → Review（验证学习成果）
         → Document（更新知识库）
```
