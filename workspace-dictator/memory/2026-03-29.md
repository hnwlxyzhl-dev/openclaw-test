### 核心工作流框架

**The Pipeline:**
```
Idea → Brainstorm → Plan → Subagent-Driven Build (TDD) → Code Review → Finish Branch
```

### Phase 1: Brainstorming（头脑风暴）
这是用户特别强调要学习的方法，用于开拓做事的思维：

1. **探索项目上下文** - 检查文件、文档、最近提交
2. **逐一提出澄清问题** - 每次只问一个问题，偏好选择题
3. **提出2-3个方案** - 包含权衡分析和推荐
4. **分段展示设计** - 按复杂度分段，每段后获取确认
5. **写设计文档** - `docs/plans/YYYY-MM-DD-<topic>-design.md` → 提交
6. **过渡到计划编写阶段**

**关键问题：**
- 你真正想解决什么问题？（目的）
- 有什么限制条件？（时间、技术栈、依赖）
- 成功的标准是什么？（成功标准）
- 这个不应该做什么？（范围边界）

**铁律：在设计被批准之前，绝不写代码！**

### Phase 2: Writing Plans（编写计划）
- 编写详细的逐任务实施计划
- 每个任务 = 2-5分钟：写测试 → 看失败 → 实现 → 看通过 → 提交
- 保存到 `docs/plans/YYYY-MM-DD-<feature>.md`

### Phase 3: Subagent-Driven Development（子代理驱动开发）
使用 `sessions_spawn` 分发任务：
1. 分发实现者子代理（含任务+完整计划上下文）
2. 等待完成通知
3. 分发规格审查子代理 → 确认代码匹配规格
4. 分发代码质量审查子代理 → 批准质量
5. 修复问题，必要时重新审查

### Phase 4: Systematic Debugging（系统性调试）
**铁律：不找到根本原因就不修复！**

四个阶段：
1. 根本原因调查（读错误、复现、检查最近变更、追踪数据流）
2. 模式分析（找工作的例子、比较、识别差异）
3. 假设+测试（一次一个假设，测试证明/反驳）
4. 修复+验证（修复根本原因，不是症状）

### Phase 5: Finishing a Branch（完成分支）
1. 验证所有测试通过
2. 确定基础分支
3. 提供4个选项：本地合并 / 推送+PR / 保留 / 丢弃
4. 执行选择

### Phase 6: TDD（测试驱动开发）
**铁律：没有失败的测试就不能写生产代码！**

Red-Green-Refactor:
1. RED - 写失败的测试
2. 验证 RED - 看它失败（必须！）
3. GREEN - 写最小代码让它通过
4. 验证 GREEN - 看它通过（必须！）
5. REFACTOR - 清理

### 关键原则
- 头脑风暴时**一次只问一个问题**
- **始终 TDD** - 先写失败的测试
- **YAGNI** - 移除不必要的功能
- **DRY** - 不重复
- **系统性优于临时性** - 特别是在时间压力下更要遵循流程
- **证据优于声明** - 验证后再宣布成功
- **频繁提交** - 每个绿色测试后提交

## 以后如何应用

当接到任务时，不要直接动手，而是：
1. 先澄清需求（问清楚要解决什么问题）
2. 探索上下文（查看相关文件和文档）
3. 提出2-3个方案供选择
4. 获得确认后再制定详细计划
5. 按计划执行，每步验证
