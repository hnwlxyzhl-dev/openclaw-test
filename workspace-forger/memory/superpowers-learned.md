# Superpowers 技能学习笔记

学习日期：2026-03-29

## 核心工作流

```
Idea → Brainstorm → Plan → Build (TDD) → Code Review → Finish
```

**硬性门槛**：设计批准前绝不写代码。"太简单不需要设计"永远是错的。

---

## Phase 1: Brainstorming 头脑风暴

### 检查清单（按顺序）

1. **探索项目上下文** — 检查文件、文档、最近提交
2. **问澄清问题** — 一次只问一个，优先多选题
3. **提出 2-3 种方法** — 带权衡分析和推荐
4. **分段展示设计** — 根据复杂度分节，每节后获得批准
5. **写设计文档** — `docs/plans/YYYY-MM-DD-<主题>-design.md` → 提交
6. **转交** — 调用 writing-plans 阶段

### 关键问题

- 你真正想做什么？（目的）
- 有什么约束？（时间、技术栈、依赖）
- 成功的标准是什么？
- 这件事不应该做什么？（范围边界）

### 设计文档要覆盖

- 架构概览
- 组件及其职责
- 数据流
- 错误处理方式
- 测试策略

---

## Phase 2: Writing Plans 写计划

### 计划文件位置

`docs/plans/YYYY-MM-DD-<功能名>.md`

### 任务粒度

每个任务 = 一个动作，2-5 分钟：

1. 写失败的测试
2. 运行测试 — 确认失败
3. 写最小实现
4. 运行测试 — 确认通过
5. 提交

### 规则

- 总是写精确文件路径
- 计划中写完整代码
- 精确命令 + 预期输出
- DRY, YAGNI, TDD，每个绿测后提交

---

## Phase 3: TDD 测试驱动开发

### 铁律

```
没有失败的测试，不写生产代码
```

写了代码才写测试？删掉重来。没有例外。

### Red-Green-Refactor

1. **RED** — 写一个最小测试展示应该发生什么
2. **验证 RED** — 运行测试，确认失败（必须！）
3. **GREEN** — 写最简单的代码让测试通过
4. **验证 GREEN** — 运行全量测试，确认通过（必须！）
5. **REFACTOR** — 清理代码（测试必须保持绿色）

### 好测试的特质

| 品质 | 好 | 坏 |
|------|-----|-----|
| 最小 | 一件事 | 名字里有"and"？拆分 |
| 清晰 | 描述行为 | `test1`, `test_works` |
| 展示意图 | 展示期望的 API | 隐藏代码应该做什么 |

---

## Phase 4: Systematic Debugging 系统调试

**硬性门槛**：没有根因调查，不修复。

### 四阶段

1. **根因调查** — 读错误、复现、检查最近改动、追踪数据流
2. **模式分析** — 找工作示例、对比、识别差异
3. **假设 + 测试** — 一次一个假设，测试证明/证伪
4. **修复 + 验证** — 从根修复而非症状；验证修复不破坏其他

---

## Phase 5: Finishing Branch 完成分支

1. 验证所有测试通过
2. 确定基础分支
3. 展示 4 个选项：本地合并 / 推送+PR / 保留 / 丢弃
4. 执行选择
5. 清理

---

## 核心原则

- **头脑风暴时一次只问一个问题**
- **总是 TDD** — 先写失败的测试，删掉测试前写的代码
- **YAGNI** — 从所有设计中移除不必要的功能
- **DRY** — 无重复
- **系统化胜过临时起意** — 越是时间紧越要遵循流程
- **证据胜过声称** — 验证后再宣布成功
- **频繁提交** — 每个绿测后提交
